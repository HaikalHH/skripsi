# WhatsApp Finance Assistant MVP (Monorepo)

MVP chatbot keuangan pribadi berbasis WhatsApp dengan AI (Gemini + Vision OCR), penyimpanan MySQL (Prisma), web customer + admin monitoring, dan chart PNG dari Python service.

## Architecture

- End-user interface: **WhatsApp + Web**
- Bot channel: **Baileys WhatsApp socket worker** (`apps/bot`)
- Backend API: **Next.js 14 route handlers** (`apps/api`)
- Web app (customer + admin): **Next.js 14** (`apps/admin-web`)
- Report renderer: **FastAPI + Matplotlib PNG/PDF** (`services/report-renderer`)
- Shared schemas/types/prompts: `packages/shared`
- Database: MySQL + Prisma

## Core MVP Features Implemented

1. WhatsApp text/image flow:
   - text: intent classification + extraction (Gemini)
   - image: OCR (Vision API) -> normalization (Gemini) -> transaction insert
2. Auto user record creation by WhatsApp number on first message (status `PENDING`); user must type `register` to start registration.
3. Transaction schema + message logging + AI analysis logging in Prisma.
4. Reporting:
   - daily, weekly, monthly, calendar, and cashflow report commands
   - aggregation in API
   - category budget progress in report text
   - chart PNG and monthly PDF generated by Python service when enabled
5. Budget/goals:
   - category monthly limits stored through `ExpensePlan` and `ExpensePlanItem`
   - `/budget set` conversational flow with `simpan`, `edit`, and `buang`
   - `/budget list` or natural category-list request
   - goal tracking through `FinancialGoal` and `GoalContribution`
   - timeline/progress commands such as `lihat timeline` and `progress goal saya`
6. Web app:
   - admin login
   - Users list/detail, Transactions audit, Reminders, Observability, and System health
7. Security/config:
   - secrets in env
   - zod validation for payloads + AI output
   - in-memory rate limiting per WA number
   - graceful fallback when AI/OCR fails (no partial transaction insert)
8. Tests:
   - extraction parsing
   - aggregation logic
9. New user onboarding:
   - unregistered users must type `register` to start registration Q&A
   - other messages from unregistered users are blocked with register prompt
   - after onboarding is complete, finance tracking and reminders are active immediately
10. Proactive reminders:
   - budget reminder (near limit / exceeded)
   - weekly spending spike reminder
   - recurring reminder
   - cashflow reminder
   - goal reminder
   - monthly closing reminder
   - bot triggers reminder sweep periodically

## Local Development (No Docker)

### 1) Install dependencies

```bash
pnpm install
```

### 2) Setup MySQL local

Create database:

```sql
CREATE DATABASE finance_bot;
```

Set `DATABASE_URL` in `apps/api/.env` to your local MySQL instance, example:

```env
DATABASE_URL=mysql://finance:finance@localhost:3306/finance_bot
```

### 3) Run DB migration + seed

```bash
pnpm --filter @finance/api prisma:generate
pnpm --filter @finance/api prisma:migrate
pnpm --filter @finance/api prisma:seed
```

### 4) Run report renderer service (Python)

```bash
cd services/report-renderer
python -m venv .venv
# Windows PowerShell:
. .venv/Scripts/Activate.ps1
# macOS/Linux:
# source .venv/bin/activate
pip install -r requirements.txt
uvicorn app.main:app --reload --port 8000
```

### 5) Run API + Web + Bot (separate terminals)

```bash
pnpm dev:api
pnpm dev:admin
pnpm dev:bot
```

### 5a) Connect WhatsApp with Baileys

Run the bot worker and scan the QR code from WhatsApp Linked Devices:

```bash
pnpm dev:bot
```

The Baileys session is stored in `BAILEYS_AUTH_DIR`.

## Sample WhatsApp Commands + Expected Response

1. `makan siang 45000`
   - expected: transaction confirmation (type/amount/category/date)
2. `gaji 5000000`
   - expected: income transaction confirmation
3. `laporan hari ini` or `/daily report`
   - expected: daily summary with category budget progress
4. `laporan minggu ini` or `/weekly report`
   - expected: weekly summary with category budget progress
5. `/monthly report`, `/calendar report`, or `/cashflow report`
   - expected: monthly summary/PDF mode depending on command
6. `/help`
   - expected: command list
7. `register` (for new/unregistered number)
   - expected: start onboarding
8. `/budget set`
   - expected: bot asks category and monthly limit, then waits for `simpan`, `edit`, or `buang`
9. `/budget list` or `lihat list kategori budget`
   - expected: saved category list from active expense plan
10. `/set goal`
   - expected: starts target creation flow
11. `/goal add` or `nabung 500rb`
   - expected: goal contribution flow/progress update
12. `/goal status`, `progress goal saya`, or `lihat timeline`
   - expected: goal progress/timeline with progress bars
13. `catat tabungan 10jt`
   - expected: portfolio deposit/cash asset created
14. `catat emas`
   - expected: guided gold asset flow
15. `catat saham BBCA 2 lot harga 9000`
   - expected: stock portfolio asset created
16. `catat properti rumah 300jt`
   - expected: property portfolio asset created
17. `portfolio`, `aset saya berapa sekarang`, or `komposisi aset`
   - expected: portfolio summary and asset detail bubbles

## New User Flow

1. User sends any message and gets prompt to type `register`.
2. User types `register`.
3. Bot asks active onboarding questions:
   - target selection and target details
   - budget mode with only two choices: manual breakdown or guided setup
   - employment and active/passive income
   - expense breakdown or guided categories
   - asset selection: tabungan, emas, saham, properti, or belum punya
   - final analysis and timeline
4. After registration complete, finance tracking, reporting, portfolio, goals, and reminders are active.

## API Notes

- Main WhatsApp ingress: Baileys worker in `apps/bot`
- Internal bot ingress used by Baileys worker: `POST /api/bot/inbound`
- Heartbeat: `POST /api/bot/heartbeat`
- Reminder sweep trigger (internal): `POST /api/bot/reminders/run`
- Bot outbound queue:
  - `GET /api/bot/outbound`
  - `POST /api/bot/outbound/ack`
- Admin endpoints (require header `x-admin-token`):
  - `GET /api/admin/users`
  - `GET /api/admin/transactions`
  - `GET /api/admin/health`

## Assumptions

1. Single MySQL database for all modules.
2. In-memory rate limit is sufficient for MVP (not distributed).
3. Admin auth is env-password + cookie session (no RBAC/JWT/OAuth yet).
4. Amount/category extraction relies on Gemini strict JSON response.
5. If OCR/Gemini/reporting fails, bot returns fallback message and does not store partial transaction data.
6. The active web UI is the admin web app in `apps/admin-web`; user-facing interaction is primarily through WhatsApp.
